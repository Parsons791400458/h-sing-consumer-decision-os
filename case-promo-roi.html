Case 02 · Promotion / Trade Fund / ROI

促销 ROI 与 Trade Fund 决策系统

这个案例不是证明“我做过 BI”，而是证明：我理解消费品企业最核心的经营议题之一——促销费用如何影响销量、利润、客户关系与资源配置，并能把这个问题转成可复盘的数据与决策系统。

100亿+促销费用决策场景
50+PEA review / training 场景
99%+Trade Fund 运作质量
企业为什么要做

为什么重要

消费品企业利润很大一部分会被促销、折扣、返利、活动费用吞掉。问题不在于“有没有投钱”，而在于：

  • 哪些促销带来了真实增量？
  • 哪些促销只是把未来销量提前？
  • 哪些客户 / SKU / 渠道吃掉了利润？
  • 哪些费用在系统里流转但很难及时复盘？
我在证明什么

证明的认知与能力

我不是停留在 dashboard 层，而是能理解促销分析背后的业务逻辑、费用机制和算法口径，并把这些逻辑沉淀到 PEA / CVE / Trade Fund 的系统能力里。

一句话：我知道消费品企业为什么必须把促销费用数字化——因为这决定利润质量，而不是单纯决定销量好不好看。

业务问题

在快消企业里，促销费用分散在多个客户、系统和活动类型中。如果没有统一的数据口径与分析系统，常见后果就是：

  • ROI 看不准，复盘靠经验而不是机制；
  • 费用核销慢，经营节奏被拖住；
  • 业务团队知道“花了很多钱”，但不知道钱到底花得值不值；
  • 渠道 / SKU / 门店层面的真实表现难以被拆解。

系统机制：把经营黑箱变成可计算模型

我参与的不是单点分析，而是把促销分析变成一套系统机制：

01

整合 GI / OPTIMA / SAP 等多系统数据

02

围绕 POS、价格、SKU、门店、客户做统一分析口径

03

拆分 Base Sales、Incremental Sales、Promotion Sales

04

让业务 review、客户利润、费用管理形成闭环

关键理解：促销数字化的核心不是做一个漂亮报表，而是把“钱花出去之后发生了什么”这件事变成可解释、可追责、可复盘的经营语言。

我能讲出的业务算法口径

  • Reference Price：用于判断是否真的发生有效促销，而不是只是常态价格波动。
  • Base / Incremental Split：把自然销售和促销带来的增量拆开，避免误判促销价值。
  • Discount Threshold：判断折扣深度是否真正足以驱动销量提升。
  • Wholesale Detection：识别批发 / 囤货造成的异常销售波动，避免把异常量误算成促销成功。

结果与组织影响

  • 推动 50+ 场 PEA review 与培训，让业务、SMO、BU 形成共同语言；
  • Trade Fund 运作质量达到 99%+；
  • 支持约 CNY 15 亿 / USD 2.34 亿 funding accrual 管理；
  • Ali PEA / CVE、JD PEA / CVE 等能力帮助业务 review 不再停留在表层销量，而能看到利润质量与促销贡献。
